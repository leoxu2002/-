# 学术文献智能管理助手 — 产品介绍文档

## 一、产品概述

学术文献智能管理助手是一款面向科研人员的**私有文献知识库检索与问答系统**。它基于模块化 RAG（检索增强生成）架构，通过 MCP（Model Context Protocol）协议对外提供服务，支持研究者将个人收藏的学术论文批量导入，然后通过自然语言进行语义检索，精准定位到具体段落甚至图表内容。

**核心定位**：将研究者从"存了找不到、找到了不精准、图表搜不到"的文献管理困境中解放出来，让论文库真正成为可检索、可关联、可溯源的活知识库。

**系统形态**：以 MCP Server 方式运行在本地，无缝对接 GitHub Copilot、Claude Desktop 等日常 AI 工具，研究者无需打开专用软件，在写作或编码过程中即可随时查询文献。

---

## 二、场景痛点

### 2.1 文献规模大、增长快，人工管理难以为继

科研人员通常需要追踪数十到数百篇相关文献，且每周都有新论文发表。以本项目内置的联邦学习领域论文为例：

- *A Hierarchical Knowledge Transfer Framework for Heterogeneous Federated Learning*
- *Acceleration of Federated Learning with Alleviated Forgetting in Local Training*
- *Auditing Privacy Defenses in Federated Learning via Generative Gradient Leakage (CVPR 2022)*
- *AutoAugment: Learning Augmentation Strategies From Data*

仅靠文件夹分类和文件名记忆，很快就会陷入"存过但找不到"的困境。

### 2.2 论文内容密度高，信息定位粒度粗

一篇学术论文通常包含摘要、相关工作、方法论、实验设计、结果分析、结论等多个部分。研究者的真实需求往往不是"找一篇论文"，而是细粒度的知识定位：

- "哪篇论文提出了用生成梯度泄漏来审计隐私防御？"
- "异构联邦学习中的知识迁移策略有哪些？"
- "数据增强策略自动搜索的核心思路是什么？"

传统工具只能定位到"文件级"，无法精准定位到"段落级"。

### 2.3 图表信息检索被完全忽略

学术论文中大量关键信息以图表形式呈现：实验对比柱状图、架构流程图、消融实验表格。研究者常常记得"看过一张关于梯度泄漏攻击的流程图"，但无法通过文字搜索定位到它。**图表与文字之间存在模态鸿沟**。

### 2.4 跨论文的知识关联难以发现

研究者需要回答的往往是横跨多篇论文的问题，如"联邦学习隐私保护的主流方法对比"。这需要系统理解各论文间的语义关联，而非孤立地存储每一篇。

### 2.5 工具割裂，工作流被频繁打断

研究者需要在 Zotero 管理文献、在 PDF 阅读器中查看论文、在写作工具中引用——三个工具来回切换，检索思路被不断打断。

## 三、现有方法局限性

### 3.1 传统文献管理工具（Zotero / Mendeley / EndNote）

| 维度 | 局限性 |
|------|--------|
| 检索能力 | 仅支持元数据（标题/作者/标签）和全文关键词搜索，无法理解语义 |
| 内容粒度 | 以"整篇论文"为管理单位，无法定位到段落/图表级别 |
| 交互方式 | 需打开独立应用、手动输入查询，无法与日常写作/编码工具集成 |
| 图表处理 | 完全不支持图表内容的检索 |
| 智能程度 | 无 AI 辅助，无法自动生成摘要/标签/关联推荐 |

**典型失败场景**：搜索"异构联邦学习中的知识迁移"，因论文标题中没有精确匹配的关键词组合而搜不到，尽管某篇论文的第 3 节正讨论这个主题。

### 3.2 通用 AI 对话工具（ChatGPT / Claude 网页版）

| 维度 | 局限性 |
|------|--------|
| 私有知识 | 无法访问你本地收藏的论文全文，只能基于训练数据中的公开信息回答 |
| 引用溯源 | 无法提供精确引用来源（哪篇论文、哪一页、哪个段落） |
| 批量处理 | 无法一次性摄取并索引数十篇论文 |
| 知识持久性 | 每次对话需重新上传文档，无法建立持久可复用的知识库 |

### 3.3 简单向量检索 RAG 方案

| 维度 | 局限性 |
|------|--------|
| 专有名词 | 纯语义检索对专有名词（如 "FedAvg"、"CVPR 2022"）精确匹配能力差 |
| 检索质量 | 单路召回，无重排序机制，Top-K 结果中噪声多 |
| 可观测性 | "黑盒"检索，无法理解为什么返回了这些结果，调优无从下手 |
| 多模态 | 不处理论文中的图表 |
| 评估体系 | 无系统性评估机制，检索质量"凭感觉" |

## 四、解决思路

### 4.1 从"文件管理"升级为"知识片段管理"

不再以整篇论文为单位，而是通过 Ingestion Pipeline 将论文拆解为自包含的知识片段（Chunk），每个 Chunk 都携带丰富的上下文元数据：

```
PDF → Markdown解析 → 语义分块 → Chunk精炼 → 元数据增强 → 向量化存储
```

每个 Chunk 包含：来源论文、章节、页码、自动生成的标题、摘要、主题标签。检索粒度从"文件级"细化到"段落级"。

### 4.2 用混合检索消除"语义 vs 精确"的两难

单一检索策略无法同时满足语义理解和精确匹配：

- **Dense Embedding（稠密检索）**：理解"知识迁移"与 "knowledge transfer" 是同一概念
- **BM25 稀疏检索**：精确匹配 "FedAvg"、"CVPR 2022" 等专有名词
- **RRF 融合**：两路结果归一化合并，兼顾查全率与查准率
- **Rerank 精排**：对融合后的候选集用 Cross-Encoder/LLM 进行深度语义打分

这是"粗排广召回 → 精排提质量"的两段式架构。

### 4.3 用图转文（Image-to-Text）消除图表模态鸿沟

针对学术论文中图表丰富的特点，采用 Vision LLM 将图片"翻译"为自然语言描述，再将描述缝合进 Chunk 正文：

```
论文中的图片 → Vision LLM 生成文字描述 → 缝合进 Chunk 正文 → 统一向量化
```

这样研究者就能通过"搜文字出图"——例如搜索"梯度泄漏攻击流程"，直接定位到 CVPR 2022 论文中描述该攻击流程的架构图。

**相比 CLIP 多模态向量方案的优势**：

| 对比维度 | 图转文（本项目） | CLIP 多模态向量 |
|---------|-----------------|----------------|
| 架构复杂度 | 复用纯文本 RAG 链路，零侵入 | 需引入图像向量库、改造检索融合逻辑 |
| 图表语义理解 | Vision LLM 可解读图表含义、提取数据点 | CLIP 只能捕捉视觉相似性，无法理解语义 |
| 检索成本 | 仅摄取阶段一次性调用 Vision LLM | 检索阶段持续有额外图像向量计算开销 |
| 渐进增强 | 未来可叠加 CLIP，无需重构核心链路 | — |

### 4.4 用 MCP 协议消除工具切换成本

研究者不需要打开专门的文献管理软件，而是在日常使用的 AI 工具中直接调用：

```
GitHub Copilot / Claude Desktop → MCP协议 → 本系统三个工具
  ├── list_collections        → 发现有哪些知识集合
  ├── query_knowledge_hub     → 语义检索相关段落（带引用来源）
  └── get_document_summary    → 获取论文概览
```

零前端开发，一次部署即可在所有 MCP 兼容工具中使用。

### 4.5 用可观测性 + 评估体系驱动持续改进

检索系统的核心难题不是"一次调好"，而是"持续保持好"：

- **全链路 Trace**：记录查询的每一步（Dense 召回了什么、Sparse 召回了什么、Rerank 如何调整排序），可在 Dashboard 中可视化
- **Ragas + Golden Test Set 评估**：建立回归测试机制，每次调整参数后自动验证检索质量是否退化

### 5.2 核心数据流

**摄取流程（Ingestion）**：

```
学术论文 PDF
    │
    ├─ 1. Loader：PDF → Markdown 解析，提取图片，插入 [IMAGE: id] 占位符
    │
    ├─ 2. Splitter：Markdown → 语义分块，保持图文关联
    │
    ├─ 3. Transform（三步增强）：
    │     ├─ Chunk Refiner：LLM 精炼去噪，确保语义自包含
    │     ├─ Metadata Enricher：LLM 自动生成 Title/Summary/Tags
    │     └─ Image Captioner：Vision LLM 为图片生成文字描述，缝合进正文
    │
    ├─ 4. Encoding：Dense Embedding + BM25 Sparse 双路编码
    │
    └─ 5. Storage：向量 + 元数据写入 ChromaDB / BM25索引 / 图片存储
```

**检索流程（Query）**：

```
自然语言查询（如"异构联邦学习的知识迁移方法"）
    │
    ├─ 1. Query Processor：查询预处理与关键词提取
    │
    ├─ 2. Hybrid Search（并行双路召回）：
    │     ├─ Dense Route：语义向量检索 → Top-20 候选
    │     └─ Sparse Route：BM25 关键词检索 → Top-20 候选
    │
    ├─ 3. RRF Fusion：两路结果按排名倒数融合 → Top-10 候选
    │
    ├─ 4. Rerank：Cross-Encoder/LLM 精排 → Top-5 最终结果
    │
    └─ 5. Response Builder：带引用的格式化响应 + 关联图片返回
```

## 六、核心能力详解

### 6.1 混合检索 + 重排：兼顾语义与精确

| 检索阶段 | 策略 | 解决的问题 | 学术文献场景举例 |
|---------|------|-----------|----------------|
| Dense 检索 | 向量语义匹配 | "词不同意同" | 搜"知识迁移"能命中 "knowledge transfer" |
| Sparse 检索 | BM25 关键词匹配 | 专有名词精确查找 | 搜"FedAvg"能精确命中 |
| RRF 融合 | 排名倒数加权 | 平衡两路结果 | 综合两路候选，避免单路漏召回 |
| Rerank 精排 | Cross-Encoder | 细微语义差异识别 | 区分"联邦学习中的隐私攻击"与"联邦学习中的隐私防御" |

### 6.2 图转文多模态：让图表可检索

针对学术论文中大量架构图、流程图、实验图表的特点，系统在摄取阶段自动处理：

1. **Loader 阶段**：从 PDF 中提取图片，在文本中插入 `[IMAGE: id]` 占位符
2. **ImageCaptioner 阶段**：调用 Vision LLM（如 Qwen-VL）生成图片描述，将描述缝合进 Chunk 正文
3. **检索阶段**：与纯文本检索完全一致，无需额外处理
4. **响应阶段**：根据命中的 Chunk 关联的 `image_id`，将原始图片一并返回

**实际效果**：搜索"梯度泄漏攻击流程" → 命中包含该架构图描述的 Chunk → 同时返回原始架构图供查看。

### 6.3 全链路可插拔：适配不同研究环境

| 可插拔组件 | 支持的后端 | 典型切换场景 |
|-----------|-----------|-------------|
| LLM | Azure OpenAI / OpenAI / Qwen / DeepSeek / Ollama | 国内用 Qwen，海外用 OpenAI，离线用 Ollama |
| Embedding | OpenAI / BGE / Ollama (nomic-embed-text) | 成本敏感用本地模型，质量优先用云端 API |
| VectorStore | Chroma / Qdrant / Milvus | 小规模用 Chroma，大规模迁移至 Qdrant |
| Reranker | None / Cross-Encoder / LLM | 调试阶段关闭，上线后开启精排 |
| Vision LLM | GPT-4o / Qwen-VL / Ollama (llava) | 中文论文用 Qwen-VL，英文用 GPT-4o |

所有切换通过 `config/settings.yaml` 一键完成，零代码修改。

### 6.4 可观测性与评估：数据驱动调优

**Dashboard 六大功能页面**：

| 页面 | 功能 | 学术文献场景价值 |
|------|------|-----------------|
| 系统总览 | 组件配置、数据资产统计 | 确认当前使用哪些模型/索引 |
| 数据浏览 | 文档列表、Chunk 详情、关联图片 | 检查某篇论文的分块效果 |
| Ingestion 管理 | 触发摄取、实时进度、删除文档 | 新论文入库、过时论文删除 |
| 查询追踪 | Dense/Sparse 召回对比、Rerank 前后排名 | 理解"为什么搜到了这篇/没搜到那篇" |
| 摄取追踪 | 摄取历史、各阶段耗时 | 定位某篇论文摄取失败的原因 |
| 评估面板 | Hit Rate、MRR、Faithfulness 指标 | 量化评估检索质量，拒绝"凭感觉"调优 |

**评估闭环**：基于 Ragas 框架，建立 Golden Test Set（标准"问题-答案-来源文档"对），每次调整参数后运行回归测试，确保检索质量不退化。

### 6.5 MCP 协议集成：融入日常研究工具链

系统通过 MCP 协议暴露三个标准工具：

| 工具名称 | 功能 | 典型使用场景 |
|---------|------|-------------|
| `list_collections` | 列举可用的知识集合 | "我有哪些文献集合？" |
| `query_knowledge_hub` | 混合检索 + 重排 | "联邦学习中缓解遗忘的策略有哪些？" |
| `get_document_summary` | 获取论文摘要与元信息 | "帮我看看那篇 CVPR 2022 的论文概要" |

所有检索结果携带**精确引用**（来源论文、页码、相关性分数），确保学术引用的可溯源性和严谨性。

## 七、典型使用场景

### 场景 1：文献调研中的快速检索

> 研究者在撰写综述时，想了解"联邦学习隐私保护"的相关工作。在 VS Code 中直接向 Copilot 提问，系统从知识库中检索到 CVPR 2022 论文中关于"生成梯度泄漏"的段落，并附带精确引用。

### 场景 2：跨论文对比分析

> 研究者想对比"异构联邦学习"与"同构联邦学习"中知识迁移策略的差异。系统通过语义检索，从两篇不同论文中分别召回相关段落，提供带引用的对比内容。

### 场景 3：论文图表定位

> 研究者记得某篇论文中有一张关于"AutoAugment 搜索策略"的流程图，但记不清是哪篇。通过搜索"数据增强策略自动搜索流程"，系统利用图转文能力，命中包含该图表描述的 Chunk，并返回原始图片。

### 场景 4：新论文快速入库

> 研究者下载了新发表的联邦学习论文，通过 Dashboard 一键导入。系统自动完成解析、分块、增强、向量化全流程，新论文即刻可被检索。

### 场景 5：检索质量持续监控

> 研究者调整了 Chunk Size 参数，通过评估面板运行 Golden Test Set 回归测试，发现 Hit Rate 从 88% 提升至 93%，确认调整有效。

## 八、技术栈

| 层次 | 技术选型 |
|------|---------|
| 通信协议 | MCP (Model Context Protocol), JSON-RPC 2.0, Stdio Transport |
| 检索引擎 | Hybrid Search (Dense Embedding + BM25 + RRF Fusion + Rerank) |
| 向量存储 | ChromaDB (默认), 支持 Qdrant/Milvus |

## 十、总结

学术文献智能管理助手的核心价值在于：**将分散的私有文献转化为可通过自然语言精准检索的活知识库，并通过 MCP 协议无缝接入研究者已有的 AI 工具链**。

它解决了研究者在文献管理中面临的五大痛点：

| 痛点 | 解决方案 |
|------|---------|
| 存了找不到 | 混合检索（语义+精确），从"文件级"细化到"段落级" |
| 找到了不精准 | 两段式架构（粗排召回+精排重排），Top-K 结果精准度显著提升 |
| 图表搜不到 | Vision LLM 图转文，"搜文字出图" |
| 工具来回切 | MCP 协议集成 Copilot/Claude，在写作/编码中直接检索 |
| 调优凭感觉 | 全链路追踪 + Ragas 评估 + Golden Test Set 回归 |
